How a request flows

auth → route → compute → store → notify
cognito
Validates the JWT and confirms which tenant the request belongs to. The custom:tenant_id claim is baked into every token at login — no user can fake or change it.
api gateway
Single HTTPS entry point for every tenant. Routes POST /tasks and GET /tasks to the same Lambda functions regardless of which company is calling.
lambda (node.js)
Decodes tenant_id from the verified JWT — never from the request body or query string. Attaches it as a mandatory parameter to every single database query.
rds postgresql + dynamodb
One shared tasks table for all tenants. Every query carries WHERE tenant_id = $1. Activity logs written to DynamoDB on every write operation.
sqs + sns + cloudwatch
Background processing for activity logs and email alerts. CloudWatch monitors Lambda errors, API latency, and RDS health across all components.
The core security rule: tenant_id is read from JWT payload['custom:tenant_id'] only — decoded inside Lambda from the signed Cognito token. Even if a user sends "tenant_id": "someone-elses-uuid" in the request body, it is completely ignored. The token is the only source of truth.
